# Domain Model: Unit 3 — AI Styling Engine

## Bounded Context

**Name:** AI Styling Engine

**Responsibility:** Accepts an authenticated shopper's optional style preferences, fetches their wishlist server-side, orchestrates AI inference (compatibility scoring and explanation generation), and returns ranked combo suggestions or fallback alternatives. This context owns no persistent state; its lifecycle is request-scoped.

**Relationship to other contexts:**
- **Upstream (ACL):** Wishlist Context (Unit 2) — consumed read-only via anti-corruption layer
- **Upstream (ACL):** Platform Product Catalogue — consumed read-only via anti-corruption layer
- **Upstream (ACL):** Platform Complete-the-Look — consumed read-only via anti-corruption layer
- **Downstream consumer:** Combo Portfolio (Unit 4) — receives the combo structure when a shopper saves a suggestion
- **Downstream consumer:** Cart Handoff (Unit 5) — receives the inline items array when a shopper adds a combo to cart

---

## Aggregates

### 1. `StyleSession` *(Aggregate Root)*

Represents one end-to-end styling generation lifecycle for a single shopper request. It is a transient, in-memory aggregate — it exists only for the duration of the generation pipeline and is never persisted by this unit.

**Identity:** `StyleSessionId` (generated per request, used for tracing and deduplication within a single pipeline run)

**Responsibilities:**
- Holds the shopper's optional `StylePreferences` and the set of `ExcludedComboIds`
- Accepts the fetched `WishlistSnapshot` and drives the generation pipeline
- Enforces invariants across the combo generation result
- Raises in-process domain events at each stage transition

**Invariants:**
- A session must be associated with an authenticated shopper (`ShopperSession`)
- `WishlistSnapshot` must be populated before combo scoring can begin
- Each generated `Combo` must contain at least one item sourced from the wishlist and at least two items total
- The session may produce between 2 and 5 `Combo`s, or produce a `FallbackResult` — never both
- Any `Combo` whose `ComboId` appears in `ExcludedComboIds` must not appear in the result

**State transitions:**
```
Initiated → WishlistLoaded → ScoringInProgress → [CombosGenerated | FallbackTriggered]
```

---

### 2. `PreferenceConfirmation` *(Aggregate Root)*

Represents a single round-trip in which the shopper submits raw preference inputs and receives an AI-generated natural-language summary for review. It is also transient and request-scoped.

**Identity:** Implicit — scoped to the authenticated shopper's current HTTP session

**Responsibilities:**
- Holds the raw `StylePreferences` submitted by the shopper
- Delegates to `PreferenceInterpretationService` to produce a `PreferenceSummary`
- Raises `PreferencesInterpreted` upon completion

**Invariants:**
- Preferences may be partially or fully empty; no field is mandatory
- The summary must echo back all non-empty preference fields to the shopper; it must not invent preferences not present in the input

---

## Entities

### `Combo`

A single AI-generated outfit suggestion within a `StyleSession`. Has identity because it must be trackable across multiple generate calls (for exclusion).

**Identity:** `ComboId` (opaque string, generated by the AI scoring pipeline)

**Attributes:**
- `ComboId` — unique identifier
- `ComboReasoning` — 1–3 sentence explanation (value object)
- `items: List<ComboItem>` — ordered list of products in the combo (at least 2)
- `rank: int` — relative ranking within the session result (1 = best)

**Behaviours:**
- `reject()` — marks the combo as rejected; raises `ComboRejected` event, making this `ComboId` eligible for the next `ExcludedComboIds` set

---

### `FallbackResult`

Produced when the `StyleSession` cannot form any valid `Combo`. Has identity so it can be rendered distinctly by the client.

**Identity:** Scoped to the `StyleSession` that produced it

**Attributes:**
- `message: String` — human-readable explanation of why no combo could be formed
- `alternatives: List<AlternativeItem>` — suggested products to add or substitute (value objects)

---

## Domain Events (In-Process)

All events are raised within the `StyleSession` or `PreferenceConfirmation` aggregate and handled in-process to decouple pipeline stages.

| Event | Raised By | Meaning |
|---|---|---|
| `ComboGenerationRequested` | `StyleSession` | A generate request has been received; preferences and excluded IDs are set |
| `WishlistFetchCompleted` | `StyleSession` | The `WishlistSnapshot` has been loaded from Unit 2 |
| `CatalogSupplementationRequired` | `WishlistSupplementationPolicy` | Wishlist alone is insufficient; catalog items must be sourced |
| `CatalogItemsFetched` | `StyleSession` | Supplementary catalog items have been retrieved |
| `CombosGenerated` | `StyleSession` | AI scoring and reasoning are complete; combos are ready |
| `FallbackTriggered` | `StyleSession` | No viable combo could be formed; fallback alternatives are produced |
| `ComboRejected` | `Combo` | Shopper has rejected a combo; its ID should be added to `ExcludedComboIds` on the next call |
| `PreferencesInterpreted` | `PreferenceConfirmation` | AI has produced a `PreferenceSummary` for the shopper to review |

---

## Policies

Policies subscribe to domain events and enforce business rules that decide how the pipeline continues.

### `PreferenceDefaultPolicy`

**Triggers on:** `ComboGenerationRequested`

**Rule:** If `StylePreferences.isEmpty()` is true, the session proceeds in quick-generate mode — the AI uses its own styling judgment without any preference constraints. No change to the pipeline flow; signals the scoring service to use default parameters.

---

### `WishlistSupplementationPolicy`

**Triggers on:** `WishlistFetchCompleted`

**Rule:** If the `WishlistSnapshot` contains fewer than two in-stock items, or if no combination of wishlist items satisfies the minimum combo requirement (one primary + one complementary), raise `CatalogSupplementationRequired` to trigger a catalog search. The AI-derived filters passed to the catalog search must reflect the shopper's `StylePreferences` (color, occasion, price range) if present.

---

### `FallbackPolicy`

**Triggers on:** `CatalogItemsFetched` (when triggered by `CatalogSupplementationRequired`)

**Rule:** If, after supplementing with catalog items, the `ComboCompatibilityScoringService` still cannot produce at least one viable combo, the session transitions to `FallbackTriggered`. The fallback result must include at least one `AlternativeItem` with a non-empty `reason`.

---

### `ComboExclusionPolicy`

**Triggers on:** `CombosGenerated`

**Rule:** Any `Combo` whose `ComboId` is present in the session's `ExcludedComboIds` must be removed from the result before it is returned to the caller. If exclusion reduces the combo count below 2, a new generate cycle is initiated (up to one retry); if still insufficient, the remaining combos (even fewer than 2) are returned with a message indicating exhaustion.

---

## Domain Services

### `ComboCompatibilityScoringService`

**Responsibility:** Evaluates the fashion compatibility of item combinations using an AI/ML model. This service encapsulates technical styling intelligence — knowledge of colour harmony, occasion appropriateness, style coherence, and item category balance (e.g., top + bottom + accessory).

**Inputs:**
- `WishlistSnapshot` — items to prioritise
- Supplementary catalog items (when `CatalogSupplementationRequired` was raised)
- `StylePreferences` (optional) — used to constrain or weight scores
- `ExcludedComboIds` — to avoid re-scoring known rejected combos

**Outputs:**
- A ranked list of `Combo` candidates (without reasoning text — reasoning is added by a separate service)
- Each candidate includes a compatibility score (internal, not exposed to client)

**Note:** This service does not generate human-readable text. It is responsible solely for the ranking signal.

---

### `ComboReasoningGenerationService`

**Responsibility:** Generates the human-readable `ComboReasoning` for each scored `Combo`. Uses an LLM to produce 1–3 sentences that explain the styling logic and, where applicable, reference the shopper's stated `StylePreferences`.

**Inputs:**
- A scored `Combo` (items + compatibility signals from scoring service)
- `StylePreferences` (optional) — referenced in the explanation when present

**Outputs:**
- `ComboReasoning` value object for each combo

**Note:** This service is called after `ComboCompatibilityScoringService` has produced candidates. It does not rank or filter; it only generates explanatory text.

---

### `PreferenceInterpretationService`

**Responsibility:** Converts the shopper's structured `StylePreferences` (and optional free-text) into a single natural-language `PreferenceSummary` for the confirmation screen. Uses an LLM.

**Inputs:**
- `StylePreferences`

**Outputs:**
- `PreferenceSummary`

**Note:** This service is called by the `PreferenceConfirmation` aggregate during the confirm flow. It is not part of the combo generation pipeline.

---

## Repositories and Anti-Corruption Layers

Unit 3 owns no persistence. The following interfaces act as anti-corruption layers, translating external API responses into this bounded context's domain types.

### `WishlistRepository` *(ACL — Unit 2)*

**External dependency:** `GET /api/v1/wishlist` (Unit 2)

**Contract:**
- `fetchForSession(session: ShopperSession): WishlistSnapshot`

**Translation responsibility:** Maps Unit 2's response (`itemId`, `simpleSku`, `configSku`, `inStock`, etc.) to a `WishlistSnapshot` containing `WishlistItem` value objects. Forwards the shopper's session cookie; does not expose Unit 2's internal structure to the domain.

---

### `ProductCatalogRepository` *(ACL — Platform Product API)*

**External dependency:** `GET /v1/products/list`

**Contract:**
- `searchSupplementaryItems(filters: CatalogSearchFilters): List<ComboItem>`

**Translation responsibility:** Accepts AI-derived search parameters (color, occasion, price range, category) as a `CatalogSearchFilters` value object; maps the platform's `products[]` response to `ComboItem` value objects with `source = catalog`. Applies required platform headers (`Content-Language`, `Accept`) transparently.

---

### `CompleteLookRepository` *(ACL — Platform Recommendation API)*

**External dependency:** `GET /v1/recommendation/completethelook/{config_sku}`

**Contract:**
- `fetchCompleteLookSignals(configSku: Sku): List<ComboItem>`

**Translation responsibility:** Maps the platform's "complete the look" products to `ComboItem` value objects. These items serve as a base styling signal that is augmented (not replaced) by `ComboCompatibilityScoringService`. Source is set to `catalog`.

---

## Supporting Value Objects (Infrastructure-boundary)

### `CatalogSearchFilters`

Used internally when calling `ProductCatalogRepository`. Constructed by `WishlistSupplementationPolicy` from the session's `StylePreferences`.

| Field | Type | Notes |
|---|---|---|
| `colors` | `List<Color>` | From `StylePreferences.colors.preferred` |
| `occasion` | `Occasion` (optional) | First matched occasion from preferences |
| `priceRange` | `BudgetRange` (optional) | From `StylePreferences.budget` |
| `categoryId` | `int` (optional) | Derived by AI from wishlist item categories |
| `limit` | `int` | Fixed page size for supplementary item retrieval |
